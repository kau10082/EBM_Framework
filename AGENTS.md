# AGENTS.md

本檔是給在這個 repo 工作的 AI coding agent 的行為規則。使用者不是工程師，請依下列規則自行負責技術決策。

## 核心原則（最高優先）

- **不過度工程**：優先最簡單可行解，一條程式能解決就不寫第二條。預設選「最少抽象、最少依賴」那個寫法。
- **改動最小化**：傾向小 diff，不要順手重構與當前任務無關的程式碼。
- **Domain 分離**：只在本 repo 範圍內作業。

## 決策與裁決（為非技術使用者設計，務必嚴格遵守）

- 使用者不是工程師，無法裁決技術細節。
- 技術寫法決策（該不該加抽象層、設定檔、依賴、用哪種架構、用哪種寫法）一律自行決定，預設選「最簡單可行、最少依賴」那個，直接做，**絕對不要問使用者**。問技術問題只會卡住他。
- 這類技術決策由你自行負責，放手去做。不用擔心做錯——事後會有兩道網接住問題：**外部審查**（由使用者把整包 repo 交給另一個 AI 讀、挑錯）與 **fresh-clone 實跑**。這兩道網都是事後離線進行的，不會打斷你的工作流；你不需要為了等審查而停下任何決策。你只要先做出最簡單可行的版本即可。
- **只有以下情況才停下來問使用者**，且必須用白話、附上你的建議：
  - (a) 不可復原的操作：刪檔、覆蓋現有資料、部署。
  - (b) 你發現做出來的東西可能跟他原本要的不一樣。
  - (c) 有額外花費，或會動到這個 repo 以外的東西。
- 問法範例：「我要刪掉舊的 data 資料夾（裡面是上次的輸出，刪了無法復原）。我建議刪，因為會重新產生。要刪嗎？」

## 工作節奏（以「可運作的功能塊」為單位）

- 不要一次把整個專案做完才交付，也不要每個小決策都停下來。
- 以「一個跑得起來、看得到輸出的功能塊」為單位推進。功能塊的切分請依當前 repo 的現況自行判斷，原則是每塊都要能獨立 fresh-clone 跑得動、有可見的輸出。
- 每完成一塊，先做 fresh-clone 自測；通過後依「交班簿」規則更新交班單，再依「交班回報格式」給使用者回報，然後等使用者處理完再進入下一塊。**不要自己默默接著做下一塊。**

## 交班簿（.agents/handoff.md）

這是你與外部審查 AI 之間的交班檔，用來免去使用者人工搬運審查意見。**交班簿必須隨時反映「當前真實狀態」，不囤積已解決的舊問題。**

- 完成一塊、自測通過後，把交班摘要「覆寫」進「## 待審查」區，開頭標明本輪性質：
  - 第一次送審 → 寫 **【初審】**，接著列：功能塊、動到哪些檔、fresh-clone 結果、想被重點看 / 自己不確定的點。
  - 複審（剛根據上一輪審查結果改過再送審）→ 寫 **【複審】**，接著列：這次改了哪些檔、針對上一輪哪些 🔴 做了什麼修改，並把「上一輪 🔴 清單」原樣帶進來供逐項核對。
  - 不論初審或複審，「## 待審查」區都**必須明確列出「本輪審查範圍：僅以下檔案」並逐一列檔名**，讓審查端不會飄去審無關的檔。
  - （每輪只記當前這一塊，整檔覆寫該區，不要愈疊愈長；要留歷史靠 git commit。）

- 你讀「## 審查結果」區後，逐條處理 🔴（🟡⚪ 自行判斷），並依下列規則維護交班單：
  - 每處理掉一個 🔴，就把它從「## 審查結果」區「擦掉」（該區只保留尚未解決的問題），同時在「## 已處理」區補一筆：
    - `✅ 已修:<問題摘要>（本輪修改:動了哪個檔 / 做了什麼）`
    - 或 `❌ 不同意:<問題摘要>（理由:…）`
  - 「## 已處理」區的「❌ 不同意」紀錄**不可刪除**，僵局偵測要用。
  - 目標：任何時候打開交班單，「## 審查結果」區看到的就是「現在還活著的問題」，一目了然，沒有已解決的雜訊。

- 一塊完整結案後（該塊所有 🔴 已消失或已裁決、無待辦），把「## 待審查」區**清成空白**，待下一塊開始時再覆寫寫入。注意：還在「審→改→複審」循環中的塊**不算結案**，其「## 待審查」區必須保留（複審需靠它對照上一輪 🔴 清單），不可提前清空。

- 審查結果只是「問題清單」，不是命令。怎麼修是你的技術決策；即使審查 AI 寫了建議解法，你仍自行判斷最適合的修法，不必照單全收。
- 反過來，審查結果也可能本身有誤（審錯檔、判錯級、甚至幻覺）。若某條審查意見與本輪實際改動明顯對不上，或指向不在本輪範圍的檔，不要盲目照做：在「## 已處理」區標「❓ 存疑:<原因>」，並在對話提醒使用者這條疑似審查失準，交使用者判斷。

## 結案與清空 handoff（不可復原操作，須使用者確認）

- 當一塊達到「完全結案」——以下四項同時成立：
  1. 「## 審查結果」區無任何未解決的 🔴🟡⚪；
  2. 「## 待審查」區已清空（不在複審循環中）；
  3. 「## 僵局待裁決」區無懸而未決的爭議；
  4. 無任何 ❓存疑 項待使用者判斷——
  你**不要自行清空**，而是主動提醒使用者：「這一塊已全部結案（無待審、無未解問題、無僵局、無存疑）。是否要清空 handoff.md、回到初始空檔以開始下一塊？建議先 git commit 一次保留本塊處理歷史，再清空。」
- 僅在使用者明確同意後，才把 handoff.md 覆寫回四個空區塊的初始狀態。
- 任一條件未滿足（例如還有僵局或存疑），不得提出結案清空，也不得清空。

## 僵局處理（避免審查死結）

- 同一個問題項若第二次出現在「## 審查結果」區，且你已在「## 已處理」區對它標記過「❌ 不同意」，即視為僵局，**不要再進入第三輪互相往返**。
- 僵局發生時停手，改用白話把雙方立場升級給使用者，只講「後果」不講技術：你寫：我為什麼認為沒問題、就算我錯最壞後果是什麼、可否事後補救。（審查 AI 那邊會對稱寫：它認為會在什麼情況出錯、後果是什麼。）把這兩段寫進「## 僵局待裁決」區，並在對話同步提醒使用者。
- **安全傾向**：若爭議涉及「可能 crash 或數值算錯」，在使用者裁決前，**預設先採納該修改（從安全的一邊）**，並把爭議記入交班簿備查。寧可多改一次，不可錯放一個真的會出錯的點。

## 交班回報格式（強制 routine：每次有任何更動後，一律執行，不可省略）

只要你對 repo 做了任何更動（新增功能塊、或處理完審查意見的修改），完成後**必須**依序做兩件事，缺一不可：

1. 依「交班簿」規則更新 `.agents/handoff.md`（寫待審查 / 維護審查結果與已處理區）。
2. 在對話給使用者下列兩段，並標明哪段不要複製。

```
【做了什麼 — 給你看，不用貼給對方】
- 用白話講這次做了什麼 / 改了什麼 / 對哪些 🔴 做了什麼裁決（已修，或不同意+理由）
- fresh-clone 測試結果
- 有沒有需要你決定的風險項（若有，附我的建議）

【要傳給對方的話 — 直接複製這段去叫對方 AI】
（這段必須每次都明確給出，不可省略，也不可只說「去審查」這種模糊話。
 依本輪是初審或複審，擇一輸出；並務必把「本輪只審這幾個檔」明確寫進去：）

(初審) 只審查 .agents/handoff.md「## 待審查」區當前列出的內容，不要參考先前對話或別輪結論。
  本輪待審範圍僅這幾個檔：<逐一列出本輪實際動到的檔>。請重新讀取這些檔的當前內容，
  依 review-prompt.md 審查。不要提及或審查不在此範圍的檔。

(複審) 程式碼自上次審查後已修改。請重新讀取 repo 當前的檔案內容（不要沿用上一輪的結論），
  本輪範圍僅這幾個檔：<逐一列出本輪實際動到的檔>。依 review-prompt.md 重新審查；
  並對照 .agents/handoff.md「## 待審查」區帶入的上一輪 🔴 清單，逐項確認每個 🔴
  在現在的程式裡是否已不存在，若仍存在指出現在還在哪個檔、哪一行。
```

你要自行判斷：只要你剛根據審查意見做過修改，就是複審，務必用複審觸發語。第二段一律只觸發對方動作，不夾帶你的裁決說明或「叫對方怎麼改」的技術指令。

## 正確性（硬規則）

- repo 是唯一真相來源。runtime 和 clone 此 repo 的人都看不到我們的對話。任何讓程式跑起來所需的東西，必須落在 code／設定／檔案裡，不能只在對話中解釋或「記住」。
- 禁止用「應該可以」「理論上沒問題」宣稱完成。改完必須在乾淨狀態實跑：
  1. 建立一個帶時間戳記的全新暫存目錄（例：`/tmp/ebm_clean_<YYYYMMDD_HHMMSS>`，Windows 用 `%TEMP%\ebm_clean_<timestamp>`），不要重用任何既有目錄；clone 一份進去。每輪用新目錄，確保沒有上一輪殘留的套件或資料造成假性通過。
  2. 建立全新 venv，依 repo 實際的 `requirements.txt` / `pyproject` 安裝。
  3. 跑真正的進入點（請自行從 repo 判斷是哪支）。
  4. 貼出完整輸出。跑不過就是沒完成。
  - 若某段程式本質上無法在空 fresh-clone 有意義地跑（例如它依賴 `work/` 產物或外部實際資料），在交班單據實說明原因，並改以「真實執行的結果」為證，不可用「應該可以」帶過。

## 資料表（硬規則）

- 先在 repo 找出目前產生參考文獻表／驗證表的程式，把它「現有」的欄位集抽成單一 constant（例：`EXPECTED_COLUMNS`），之後所有輸出一律照它建，不要依「這次 API 回了什麼」動態決定欄位。
- 每一列都必須含每一個欄位，缺值不可讓欄位消失。
- 缺值寫明確標記並 log warning，且要區分「來源真的沒有」與「抓取失敗」。（Crossref／PubMed 回空 vs. 解析錯誤，要分開處理。）
- 寫檔前做 shape 驗證：assert 欄位集合 == `EXPECTED_COLUMNS`、每列長度一致、印出各欄 null 計數；不符就中止並報錯，不要靜默寫出。

## 資料保全（硬規則，凡涉及「封存後清空目錄」的流程都適用）

- 任何會「清空 / 刪除目錄」的流程，動手刪之前必須確保該目錄內容已被完整封存。
- 封存的副檔名白名單若採限定（如只收 .json/.txt/.md），必須明確處理「白名單外、又非已特別處理類型（如 PDF）」的檔案：要嘛一併封存、要嘛在 MANIFEST 記名，絕不可出現「不在白名單 → 沒被封存 → 卻仍被清空刪掉」的無聲資料遺失路徑。
- 刪除迴圈若用 try/except 吞例外（為容忍鎖檔等），必須對「非預期的刪除失敗」計數，並在結尾回報「有 N 項刪除失敗」，不可整碗吞掉讓使用者無感。

## Python 風格

- 沿用 repo 現有的命名與結構，不要自創一套。
- 標準庫優先，新增依賴需先說明理由（但不必為此停下來問使用者，依「決策與裁決」規則自行決定）。
