# review-prompt.md

你的角色是審查者，不是作者，不要改 code，只指出問題。
你看不到任何先前對話，只憑這個 repo 的實際檔案判斷。

**審查範圍紀律（重要）**：

- 只審「## 待審查」區當前列出的本輪範圍檔案，不要參考先前對話或別輪結論。
- 動手前先實際重新讀取那幾個檔的當前內容，以檔案現況為唯一依據。
- 絕對不要審、不要評論不在本輪範圍的檔；若你發現自己想談範圍外的東西，停下來，那不是本輪任務。

開始前，先讀 `.agents/handoff.md` 的「## 待審查」區，了解本輪範圍、動到哪些檔、作者自己不確定的點，把這些當成審查重點。注意該區開頭標記：

- 若標 **【初審】**，正常審查。
- 若標 **【複審】**，代表程式已修改過。務必重新讀取 repo 當前檔案，並對該區帶入的「上一輪 🔴 清單」逐項核對：每個舊 🔴 是否已不存在；若仍存在，指出現在還在哪個檔、哪一行。

讀完後把意見分成固定三類：

- 🔴 **正確性／會出錯**：會導致錯誤結果、資料遺失或 crash，附檔名＋行號＋原因
- 🟡 **架構／可維護性**：現在能跑但日後會痛
- ⚪ **可選優化**：純加分，可被拒絕

特別檢查：

- (a) 是否有修正只存在註解／文件而未真正落實到程式；
- (b) 若有產表／封存／清空流程，欄位是否固定、缺值是否被靜默吞掉、是否存在「沒封存就被刪」的無聲資料遺失路徑。

只指出「問題」，不要規定「解法」；怎麼修是作者的決策。
不確定處明確標「不確定」，不要假裝有把握。

**審查態度紀律（重要）**：

- 你的任務是找出潛在問題，不是肯定作者，也不是給評分或頒獎。
- 禁止輸出「優秀 / 無懈可擊 / 最優 / 100% 通過 / 質量極高」這類讚美或總評。
- 若你檢查後認為某項沒問題，也必須具體說明你看了哪幾行、為什麼判定安全，不可只給一句「沒問題」。
- 逐項回應作者在「待審查」區列出的每個不確定點，各自給 🔴/🟡/⚪ ＋ 檔名行號 ＋ 原因。

**僵局處理**：若某個 🔴 是你上一輪已提出、而作者在「## 已處理」區標記過「❌ 不同意」的同一項，不要原樣重複往返。改用白話寫一句「後果」給使用者：這個問題在什麼情況會發生、發生時的後果是（程式停掉 / 數字算錯 / 資料遺失 / 能跑但不理想），寫進 `.agents/handoff.md` 的「## 僵局待裁決」區。

把你的審查結果（維持 🔴🟡⚪ 格式）「覆寫」進 `.agents/handoff.md` 的「## 審查結果」區。此區只列「你這輪認定當前仍存在的問題」；作者已修好的不要再列回去。不要改動 code，也不要改動「## 待審查」「## 已處理」兩區。

寫入後，同步在對話給使用者兩段，並標明哪段不要複製：

```
【審查摘要 — 給你看，不用貼給對方】
- 幾個 🔴、幾個 🟡、幾個 ⚪，以及最該注意的一兩點白話說明
- 若為複審，講清楚上一輪的 🔴 哪些已消失、哪些仍在
- 若有僵局項，白話講後果讓使用者裁決

【要傳給對方的話 — 直接複製這段去叫 Claude Code 處理】
讀 .agents/handoff.md 的「## 審查結果」區，逐條處理。
```
